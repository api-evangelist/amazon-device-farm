<p style="text-align: left;"><em>This post was written by Nikhil Dabhade, Cloud Solutions Engineer, AWS Device Farm</em></p> 
<p>A good test report is essential to proper analysis of mobile app testing. It is not only the summary of test results, but also artifacts like screenshots, videos, crash reports, logs, and app-generated files. A well formed test report provides useful insights, leads to faster resolution of issues, and helps teams focus on the right priorities. With so many great testing frameworks available, with their own strengths and weaknesses, the configuration capabilities for reports are important.</p> 
<p><a href="https://aws.amazon.com/device-farm/" rel="noopener noreferrer" target="_blank">AWS Device Farm</a> provides comprehensive test reporting that includes all of the standard capabilities: screenshots, videos, and crash reports. We also know that developers need access to custom reports and artifacts that are generated by the test framework. Today, we are providing access to any custom artifact (log file, test framework generated file, or application generated file) when accessing the test results within AWS Device Farm.<br /> <span id="more-1255"></span></p> 
<h2>How to access custom artifacts in AWS Device Farm</h2> 
<h3>Upload your app</h3> 
<p>Upload your app by using the AWS Device Farm console or the AWS CLI.</p> 
<p><img alt="" class="aligncenter size-full wp-image-1256" height="852" src="https://d2908q01vomqb2.cloudfront.net/0a57cb53ba59c46fc4b692527a38a87c78d84028/2017/09/17/Step-1.png" width="1292" /></p> 
<h3>Update your test code to configure the path to custom artifacts</h3> 
<p>In your tests, configure the comma-separated path on the host machine where you want the custom test artifacts to be stored.</p> 
<p>Suppose that your base use case is to take a screenshot in a test and save this screenshot. Next, we explore how to configure tests so that you can retrieve this artifact on AWS Device Farm.</p> 
<p><strong>Best practice for Appium TestNG and other test frameworks:</strong></p> 
<p>You can set the path to save a screenshot of the device screen to the default path $WORKING_DIRECTORY in your test code.&nbsp; The sample code snippet looks like this:</p> 
<pre><code class="lang-java">public boolean takeScreenshot(final String name) {
    String screenshotDirectory = System.getenv("WORKING_DIRECTORY");
    File screenshot = ((TakesScreenshot) driver).getScreenshotAs(OutputType.FILE);
    return screenshot.renameTo(new File(screenshotDirectory, String.format("%s.png", name)));
}</code></pre> 
<p>The result is that AWS Device Farm stores the screenshot at $WORKING_DIRECTORY on the host machine where the tests are executed. You can use almost any path on the host machine with a few exceptions (and they are explained in the console tool tip).</p> 
<p>We recommend using the environment variable $WORKING_DIRECTORY in your test code. Your code can be in any language and framework that AWS Device Farm supports.</p> 
<h3>Build and package your tests</h3> 
<p>Package the test, using the instructions in <a href="http://docs.aws.amazon.com/devicefarm/latest/developerguide/test-types-android-appium-java-testng.html" rel="noopener noreferrer" target="_blank">Working with Appium Java TestNG for Android and AWS Device Farm</a>.</p> 
<h3>Upload your tests to AWS Device Farm</h3> 
<p>Upload the .zip package you just generated&nbsp;to AWS Device Farm.</p> 
<p><img alt="" class="aligncenter size-full wp-image-1257" height="699" src="https://d2908q01vomqb2.cloudfront.net/0a57cb53ba59c46fc4b692527a38a87c78d84028/2017/09/17/Step-4.png" width="1253" /></p> 
<h3>Choose the devices against which you want to run your tests</h3> 
<p>Select the devices against which you want your tests to run.</p> 
<p><img alt="" class="aligncenter size-full wp-image-1259" height="829" src="https://d2908q01vomqb2.cloudfront.net/0a57cb53ba59c46fc4b692527a38a87c78d84028/2017/09/17/Step-5.png" width="1250" /></p> 
<h3>Configure your test run</h3> 
<p>On the <strong>Specify device state</strong> page, configure the path on the host machine where you want the screenshot to be saved. If you had an artifact that gets saved on the device, you can specify that path as well. The following image shows how to specify both of these paths.</p> 
<p><img alt="" class="aligncenter size-full wp-image-1260" height="1158" src="https://d2908q01vomqb2.cloudfront.net/0a57cb53ba59c46fc4b692527a38a87c78d84028/2017/09/17/Step-6.png" width="1251" /></p> 
<h3>Schedule your test run</h3> 
<p>On the <strong>Create a new run</strong> page, review and confirm your settings. When ready, choose <strong>Confirm</strong> <strong>and start run</strong>.</p> 
<p><img alt="" class="aligncenter size-full wp-image-1261" height="1077" src="https://d2908q01vomqb2.cloudfront.net/0a57cb53ba59c46fc4b692527a38a87c78d84028/2017/09/17/Step-7.png" width="1245" /></p> 
<p><strong>Retrieve the artifacts</strong></p> 
<p>After your tests are complete, you can retrieve your custom artifacts through the AWS Device Farm console from the <strong>Files</strong> section for the <strong>Teardown</strong> test of your test suite as shown in the following image.</p> 
<p><img alt="" class="aligncenter size-full wp-image-1262" height="220" src="https://d2908q01vomqb2.cloudfront.net/0a57cb53ba59c46fc4b692527a38a87c78d84028/2017/09/17/Step-8.png" width="442" /></p> 
<p>&nbsp;</p> 
<h3>Access downloaded artifacts locally</h3> 
<p>To access the artifacts from the test, choose <strong>Customer Artifacts</strong> and download the .zip file. The contents of the .zip file are shown next.</p> 
<p><img alt="" class="aligncenter size-full wp-image-1263" height="61" src="https://d2908q01vomqb2.cloudfront.net/0a57cb53ba59c46fc4b692527a38a87c78d84028/2017/09/17/Step-9.png" width="256" /></p> 
<h2>Conclusion</h2> 
<p>We described how to configure the path where you store the custom test artifacts in your test code and how to provide the same path to AWS Device Farm. This enables AWS Device Farm to retrieve these artifacts from the specified location on the host machine and the device under test.</p> 
<p>With this feature, you have easy access to any type of file produced by your test code and app. This gives you the ability to augment your test reports with relevant artifacts. You can also use these to store files like videos, screenshots, excel files, xml files, custom logs and binary files.</p>
